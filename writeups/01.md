## はじめに — 円周率を、FPUの無い石で

大学の授業で「数値積分で円周率を求めよう」という回を、私も習った覚えがあります。定番の例がこれでした。

$$
\int_{0}^{1} \frac{4}{1+x^2}\,dx = \pi
$$

右辺がきっかり $\pi$ になる。だから左辺を数値積分で近似すれば、円周率が出る。並列計算（MPI）の教科書でも、いちばん最初に出てくる「Hello, world」みたいな題材です。

このシリーズでは、その円周率を **Z80** ——私の世代でいう「ゼッパチ」、1970年代の 8 ビット CPU——に計算させます。最終的には Z80 を**何台もつないで並列**に解き、「分担して速くなる/ならない」を実測まで持っていきます。が、いきなり並列の前に。

第1回は地味に、**1 台の Z80 で、ちゃんと π を出す**ところまで。並列の話は次回からですが、ここで作る計算カーネルが、あとの回すべての土台になります。私がつまずいた用語は、その都度すこし補足を入れておきます。気楽に読んでもらえたら、と思います。

:::message
**この記事の到達点**：8 ビット・浮動小数点なしの Z80 で `∫₀¹ 4/(1+x²)dx` を台形公式で計算し、`π ≒ 3.1414` を出す。使うのは無料ツールだけ。最後に、自分の手元で再現する手順も置きます。
:::

## 背景 — 台形公式と「小数の無い世界」

まず道具立てを、一般の方にも分かるように。

**台形公式（trapezoidal rule）**：曲線の下の面積を、細長い**台形**にスライスして足し合わせる近似法です。区間 $[0,1]$ を $N$ 個に刻み、各スライスを台形とみなして面積を合計する。$N$ を増やすほど真の面積（＝積分値）に近づきます。難しくありません。「細かく刻んで台形で埋める」だけです。

問題はここから。Z80 には **FPU（浮動小数点演算装置）が無い**——というより、当時の 8 ビット CPU に小数を直接扱う仕組みはありません。`3.14` のような数を、CPU は素では知らないのです。

:::message
**FPU って？**：`1.5 × 2.3` のような小数の掛け算を 1 命令でこなす専用回路。いまの CPU には当たり前に載っていますが、Z80 の時代は「無いのが普通」でした。
:::

無いなら、どうするか。古くからの定石が **固定小数点（fixed-point）** です。考え方はシンプルで、「整数の中で、小数点の位置を自分で決め打ちする」。たとえば 16 ビットの整数のうち、上位を整数部・下位 14 ビットを小数部と**勝手に約束**する。これを **Q14** と呼びます。`3.14` は「`3.14 × 2^14` を四捨五入した整数」として持つ。足し算は普通の整数の足し算でいい。掛け算のあとに桁をずらして帳尻を合わせる。それだけで、FPU 無しでも小数の世界に入れます。

:::message
**Q14（Q形式）**：整数の下位 14 ビットを小数部とみなす固定小数点の約束。`1.0` は `2^14 = 16384`、`3.14` は約 `51445`。「整数だけど、頭の中では 14 ビット下に小数点がある」と思えば OK。
:::

道具は **SDCC**（Small Device C Compiler、Z80 用の C コンパイラ）と、同梱の Z80 シミュレータ **`ucsim_z80`**。どちらも無料で、入れるのに数十秒です。実機の Z80 は要りません。

## やってみる — そして最初の壁

C で素直に書けば、固定小数点の積分くらいすぐだろう……と最初は 64 ビット整数（`long long`）で書きました。被積分関数 $4/(1+x^2)$ には割り算が要る。精度を稼ぐため広い桁で持とう、と。

これが**一つ目の罠**でした。

:::message alert
**罠①：8 ビット CPU の 64 ビット演算は「ものすごく高い」**
Z80 には掛け算命令すら無く（除算も無い）、`long long` の四則演算はすべて**ライブラリ関数の呼び出し**に化けます。32 ビットでも重いのに、64 ビットは桁違い。シミュレータ上で延々終わらない。**「8 ビットで重い処理がどれだけ高くつくか」を、身をもって知る**ことになりました。
:::

ここで設計判断です。**精度を 1 段あきらめ、すべての途中計算が 32 ビットに収まる Q14 に作り直す**。$4/(1+x^2)$ の割り算も、分子の桁を詰めて 32 ビットに収まる形に整理する。`long long` を一掃したら、シミュレータ上であっさり走るようになりました。

「精度 ↔ 語長 ↔ 速度」は綱引きです。8 ビットでは特にきつい。**この綱引きそのものが、この題材の面白いところ**だと思っています。

もう一つ、地味だけど本質的な壁がありました。**Z80 には画面も UART も無い**ので、計算結果を `printf` で出す——その「出口」を自分で用意しないといけません。シミュレータが用意している小さな出口（simif）に文字を流し込む数行を書いて解決しましたが、ここで二つ目の罠を踏みます。

:::message alert
**罠②：crt0 のスタックが、メモリ上のデバイスを踏む**
C プログラムは `main` の前に **crt0**（スタートアップコード）が走り、スタックポインタを初期化します。SDCC の既定 crt0 は「スタックをメモリの一番上から下へ伸ばす」。出口デバイスをメモリ上に置くと、`printf` が深く再帰した瞬間、伸びてきたスタックがデバイスを**踏み潰す**。出口を**メモリではなく I/O ポート空間**に逃がして回避しました。実機の UART を I/O に置くのと同じ理屈です。
:::

:::message
**crt0 / スタックって？**：`crt0` は `main()` が呼ばれる前に動く「下ごしらえ」コード。**スタック**は関数呼び出しの戻り先や局所変数を積む領域で、ふつう高い番地から下へ伸びます。「下へ伸びる」性質を忘れると、低くない番地に置いたものを壊します。
:::

## 結果 — 8ビットのπは、何桁合うのか

`N = 64`（台形 64 枚）で走らせた結果がこれです。

```text
pi~=3.1414 (Q14, N=64)
```

真の値は $\pi = 3.14159\ldots$。誤差はおよそ `0.0002`。**8 ビットの石が、台形 64 枚で円周率を 4 桁近く当てた**。素直に、ちょっと感動します。

シミュレータは Z80 のクロック数も教えてくれます（手元の SDCC 4.5.24 +
ucsim_z80 0.9.9、`N=64`）。

```text
Total time since last reset = ... (1,311,070 clks)
Inst= 96,075
```

約 **131 万クロック**、約 **9.6 万命令**で π が 1 つ出る。これが「8 ビット
の計算は高い」の正体で、後の回で並列化したときの比較の基準値になります。
（本シリーズの絶対クロックは上記ツールチェーン＝ SDCC 4.5.24 +
ucsim_z80 0.9.9 での実測です。**同じ版なら clone してそのまま同じ値**が
出ます。別の版でも動きますが、この絶対クロックは数 % ずれます——
clone して確かめてほしいのは個々の数字ではなく**形**のほうで、台数を
増やしたときに並列で頭打ちする・しない、②が決定論で①が走るたびに
振れる、という構造は版に依りません。）

ここで一つ、私が面白いと感じた観察を書いておきます。**精度の頭を押さえているのは $N$ ではなく Q 形式（語長）の方**でした。この被積分関数は素直で台形公式の収束が速く、$N=64$ で既に丸め誤差と同じくらいの精度に達しています。$N$ をいくら増やしても、固定小数点の刻み幅以上には細かくならない。**「刻みを増やせば精度が上がる」が、ある所で語長の壁に頭打ちになる**——数値計算の授業で言葉で習うことが、目の前の数字で起きます。

## 詰まった所と、そこから得たもの

罠①②のほかに、もう一つ、教訓として強いものを。

最初、出口デバイスを `*(volatile unsigned char *)0x4000` というキャスト経由で叩いていました。`volatile` を付けたのに、**出力が一切出ない**。生成されたアセンブリを読んで判明したのが、これ。

:::message alert
**罠③：同じ番地への連続書き込みが、最適化で消えていた**
SDCC（の z80）は、キャスト越しの「同じ番地への反復ストア」を冗長と見なし、**最後の 1 個だけ残して消して**いました。`volatile` が効いていなかったのです。正しい書き方は、I/O ポートを表す `__sfr __at(...)` のような**専用の宣言**で「これは副作用だ」とコンパイラに明示すること。**生成コードを自分の目で読まないと、絶対に気づけない**たぐいのバグでした。
:::

脅しのつもりはありません。私にとっては、むしろ良い練習になりました。「`volatile` を付けたから大丈夫」と思い込まず、**コンパイラが実際に何を吐いたかを読む**——その習慣の大切さが、8 ビットの小さな世界だと、私には一発で腹に落ちました。

## ほかの解き方も、同じ道具で置いておく

台形公式が動いたので、**まったく同じ Q14 の道具立て**で、あと二つ解法を用意しました。並列の話に入る前に、引き出しを増やしておく感じです。

- **シンプソン公式**：台形が各スライスを「直線」で結ぶのに対し、2 スライスを 1 組にして「放物線」で結ぶ。同じ刻み数でも精度が桁違いに上がります（誤差が刻み幅の 4 乗で減る。台形は 2 乗）。実際、シンプソンは $N=4$ で、台形の $N=256$ 相当の精度に届きました。「細かく刻む」より「いい公式を選ぶ」ほうが、語長の限られた 8 ビットでは効く——というのを数字で見られます。
- **ニュートン法（＝ヘロンの平方根）**：$x_{n+1}=\frac{1}{2}\left(x_n+\frac{S}{x_n}\right)$ という有名な漸化式で平方根を詰めます。`1.0 → 1.5 → 1.41662 → 1.41418`（√2）と、前の値から次の値が出てくる。

この三つを並べたのには理由があります。台形とシンプソンは「区間ごとに独立」なので、後の回で**素直に並列化できます**。一方ニュートン法は「前の答えが無いと次が計算できない」鎖の形——これが**並列にしても速くならない**題材になります。その対比は最終回（第4回）で、実測の数字とグラフでやります。ここでは「同じ道具で三者が動く」ところまで。

なお、台形の計算カーネル（被積分関数と台形和）は `pi_kernel.h` という
**たった 1 ファイル**に置いてあり、この単一ノード版・第2回の分散メモリ版・
第3回の共有メモリ版は、その同じファイルを `#include` しているだけです
（`grep -rn pi_kernel.h` で 1 個しか無いことを確認できます）。違うのは
「部分和をどう束ねるか」だけ。だから後の回で π が方式・台数を変えても
**ビット単位で同じ値**になります——これは第3回以降の生命線になる性質です。

## まとめと、次回

- FPU の無い 8 ビット Z80 でも、固定小数点（Q14）と台形公式で円周率は計算できる。`π ≒ 3.1414`。
- 速度・精度・語長は綱引き。`long long` は 8 ビットでは現実的でなく、32 ビットに収める設計判断が要る。
- 精度の壁は刻み数 $N$ ではなく**語長**だった——数値計算の定理が、数字で見える。
- `volatile` を信じきらず**生成アセンブリを読む**。crt0 とスタックの伸びる向きを忘れない。

ここまでで「1 台で π を出す」土台ができました。次回（第2回）は、この計算を **Z80 を何台もつないで分担**します。各台が一部分を計算し、結果を持ち寄って合算する——いわゆる**メッセージパッシング**型の並列です。「分けたら速くなるのか？　どこまで？」を実測します。

## 再現手順（手元で動かす）

学生・教育用途でそのまま動かせるよう、コードと手順はこのリポジトリの README に「再現マニュアル」としてまとめてあります。最小の流れ：

1. SDCC（Z80 対応版）を入れる。`sdcc` と `ucsim_z80` が使えれば OK。
2. `methods/` で `make run`。`trapezoid: pi~=3.1416` などが出れば成功。
3. クロック数は、停止後にシミュレータの `state` で確認。
4. `make N=4 run-trapezoid run-simpson` と刻み数を変えてみる。

$N$ を変えると、ある所から台形の精度が伸びなくなる（語長の壁）、そしてシンプソンは早々に同じ床に達する——のを自分の目で確かめられます。小さな実験課題にちょうどいいかもしれない、と個人的には思っています。次回以降の①②も、各ディレクトリで `make run` するだけです。
