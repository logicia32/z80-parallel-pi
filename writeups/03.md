## はじめに — もう一つのやり方を、土台から作る

[第2回](./02.md) は「各台が自分のぶんを計算して、結果を**送って持ち寄る**」メッセージパッシング型でした。速くはなったが理想に遠く届かず、しかも走るたびに大きく振れた、という結果。

並列にはもう一系統あります。**皆で 1 つのメモリを共有する**型。今回はこちらを、シミュレータごと自分で作って試します。計算カーネル（`pi_kernel.h`）は第1回とまったく同じ 1 ファイルのまま、束ね方だけを変える回です。少し作るものが多いので、肩の力を抜いて読んでもらえたら、と思います。

:::message
**この記事の到達点**：N コアが 1 つの共有 RAM を取り合う自作マルチコア環境を組み、第1回と**同じカーネル**で円周率を計算。ロックを忘れると答えが静かに狂う事故（ロストアップデート＝並行処理の最重要事項）を**実測の数字**で再現し、第2回でできなかった**本物の割り込み駆動受信（IM2・手書き ISR）**を、テスト付きで動かします。
:::

## 背景 — 共有メモリ型と、Z80 の素の事情

**共有メモリ型**は、複数の CPU が同じ RAM をのぞき、そこへ書き込むことで協調します。絵にすると「**バス調停役（アービタ）＋共有 RAM＋各 CPU の専用 RAM**」。第2回のように送受信するのではなく、共有 RAM 上の同じ番地を**直接 read-modify-write する**。今回その共有アキュムレータは、共有 RAM 窓（`0xC000`）に置いた 1 個の 32bit 値で、各コアはそこをバス調停器越しに読んで足して書き戻します（スタックは専用 RAM 側 `0xBF00` なので共有窓とは衝突しません）。

:::message
**共有メモリ型 / バスアービタ**：複数 CPU が 1 本のバスで同じメモリにつながる方式。同時アクセスは衝突するので、**アービタ**（調停役）が「いま使っていいのは誰か」を 1 つに絞る。いまのマルチコア CPU の祖先にあたる考え方。
:::

ところが Z80 という石は、もともと「対称マルチプロセッサ」を想定して設計されていません。複数の Z80 が対等にバスを取り合う仕組みは、CPU 単体には無いのです。当時の実機も、デュアルポート RAM や外付けの調停回路といった**外部のハードウェア**でこれを実現していました。

なので今回は、**「N 個の Z80 コア＋共有 RAM＋バス調停＋割り込み」を自分でエミュレータ側に作る**ことになります。CPU の命令セットを一から書くのは無謀（し、車輪の再発明）なので、そこは**既存の出来あいの Z80 コアを再利用**し、自作するのは「つなぎの層」だけ——という方針にしました。足りない所だけ作る、というのは普段の仕事と同じやり方です。

## やってみる — 出来あいを使う、ということの現実

ここが今回いちばん「現場っぽい」話です。

最初に選んだ既存 Z80 コアは、取得した版で**ソースとヘッダが噛み合っておらず**、軽い手書きアセンブリは動くのに、SDCC が吐く**重い C コード（32 ビットの割り算ライブラリなど）になると破綻**しました。原因を切り分けるのに、命令を 1 つずつ追って実行を観察し、ようやく「これはこのコア側の問題で、こちらのつなぎ込みのバグではない」と確信できました。

:::message alert
**罠：出来あいの OSS は、最新版を無条件に信用しない**
個別ファイルを寄せ集めると、版がちぐはぐになることがあります。決め手は「軽い処理は動くのに重い処理で壊れる」「観察すると我々の層は無実」。**HEAD を信じすぎず、文書化された“縫い目”で隔離する**——再利用の鉄則を、身をもって確認しました。
:::

判断としては、**単一ヘッダで完結する、実績のある別の Z80 コアに乗り換え**ました。単一ファイルなら「版のちぐはぐ」が構造的に起きません。つなぎ込みの境界は、書いたあとに腰を据えて読み合わせ、思い込みを潰してから先へ進めました。結果、第1回の重い固定小数点カーネルが、ちゃんと完走するようになりました。「どの部品を信じ、どこを自分で握るか」——再利用の判断そのものが、この回の中身です。

もう一つ地味な罠。第1回でも触れた crt0（起動コード）が、スタックを共有メモリの窓に伸ばしてしまい、複数コアのスタックが互いに踏み合う。起動コードのスタック設定を各コア専用領域へ向け直して回避しました。第1回の罠の、マルチコア版です。

## ロックを忘れると、答えが静かに狂う

ここが並行処理でいちばん大事な事故です。各コアは共有 RAM の合計値を
**読み**、自分のぶんを**足し**、**書き戻す**。この read–modify–write の
途中で別のコアが割り込むと、後から書いたコアが、前のコアの加算ごと
**古い値で上書き**してしまう。前のコアの貢献が消える＝ロストアップデート。

全コアをバリアで一度集合させ、せーので共有合計を更新する。**ロック無し**
で実測すると（自作シミュレータは決定論的なので、何度走らせてもこの値）：

```text
ロック無し  P=2 : 共有合計=1349508  → π=1.28699
            P=3 : 共有合計=1106018  → π=1.05475
            P=4 : 共有合計= 988371  → π=0.94257   （正解は 3.14148）
```

`0` ではありません。**派手なゼロでも、それらしい 3.14 でもなく、中途半端に
小さい値**——これがロストアップデートの本当の顔です。各コアがほぼ同時に
「まだ誰も足していない古い合計」を読み、自分のぶんだけ足して書き戻すので、
**生き残るのは実質いちばん最後に書いたコアの貢献に近い**ものだけ。だから
正解の 3 分の 1 前後に落ちる。台数で値が変わるのも「どの順で潰し合うか」
が変わるから。授業で習う「lost update」が、3.14 でも 0 でもない**過小集計**
として、これ以上ないほど具体的に見えます（前の版ではここを「必ず 0／
全壊」と書いていました。実測すると 0 にはなりません。直します）。

:::message
**ロストアップデート / 排他制御**：複数の実行主体が同じデータを同時に更新すると、片方の更新が消える事故。防ぐには「同時に 1 人しか触れない」仕組み＝**ロック（排他制御）**が要る。
:::

そこで**ハードウェアセマフォ**——「いま臨界区間に入っていいのは 1 コアだけ」を保証する札——を入れます（実機の調停付きデュアルポート RAM が持っていた HW ミューテックスと同じ発想で、I/O ポートの test-and-set として実装）。札を取れたコアだけが共有 RAM の合計を更新し、終わったら返す。

```text
ロック有り  P=1..4 : 共有合計=3294113  raw=51470  π=3.14148
            spin(競合待ち) P=2:9  P=3:23  P=4:46
```

ロックを入れると、**第1回・第2回と同じカーネル**（同じ `pi_kernel.h`。
束ね方が違うだけの別 binary）が、台数によらず**ビット単位で同じ答え**
`3.14148` を出しました。しかも spin が P とともに 9→23→46 と増える＝
コアたちは確かに札を奪い合っていて、それでも正しい。これが排他制御の
効きめです。

## できなかったことが、できる — 本物の割り込み

第2回でポーリングに甘んじた**割り込み駆動の受信**を、今回は本物で実装し、
**`make run` で実走するテスト付き**で示します。周辺デバイス（NIC 役の
ホスト）が割り込み線 `/INT` を上げ、Z80 は **IM2（ベクタ割り込み）** で
`I` レジスタ＋ベクタから飛び先表（`0x0200`）を引き、手書き ISR へ飛び、
用が済んだら `EI` してから `RETI` で戻る。

:::message
**IM2 ベクタ割り込み**：Z80 の割り込みモードの一つ。デバイスが「自分の番号」を出すと、CPU が `I`レジスタ×256＋番号 の表を引いて対応ルーチンへ自動で飛ぶ。Z80 ファミリの周辺チップが連携する、当時としては洗練された仕組み。
:::

メインループは NIC を**一切ポーリングしません**（ソースを読むと、main は
ISR が立てるフラグだけを見ていて、NIC ポートには触れていません）。バイトは
割り込みだけが運んでくる。ホスト側ハーネスは届けたバイト数と、CPU が
**実際に受理した割り込み回数**、受信チェックサムの 3 つを突き合わせて
PASS/FAIL を出すので、「動いた体」では通りません。実測：

```text
IRQ DEMO (IM2, hand-written ISR):
  bytes delivered by interrupt : 4
  interrupts actually taken    : 4
  checksum  expected=0xAA  z80-reported=0xAA
  -> PASS (received only via the IM2 ISR)
```

ここで 1 つ、本物を作ったからこそ踏んだ罠を。chips コアの IM2 は割り込み
受理時に割り込みを自動で禁止し、`RETI` だけでは**再許可されません**。
かといって ISR の頭で先に `EI` すると、レベル検出の `/INT` がまだ下りて
おらず即再入＝割り込みストーム。**順番が命**で、ISR は「まず NIC を読んで
（これで `/INT` が下りる）→ それから `EI` → `RETI`」。この“レベル割り込み
ISR の作法”は、手書きで土台を持って初めて手触りでわかりました。これが
最終回の比較材料（割り込み駆動 vs ポーリング）になります。

## まとめと、次回

- 共有メモリ型を土台から自作。CPU コアは出来あいを再利用し、つなぎだけ自作（足りない所だけ作る）。
- 出来あいの OSS は HEAD を盲信しない。版のちぐはぐは「軽いと動く・重いと壊れる」で見抜く。
- **ロックを忘れると答えは 0 ではなく“過小集計”に静かに狂う**（P=4 で π≈0.94、決定論的に毎回同じ）。ハードウェアセマフォで第1・2回と**同じカーネル・ビット一致**の `3.14148`。
- 第2回でできなかった**本物の割り込み駆動受信（IM2・手書き ISR）**が、テスト付きで動く。レベル割り込み ISR は「読む→EI→RETI」の順が命。

次回（最終回）は、第2回（送って持ち寄る）と第3回（共有して取り合う）を、**同じカーネル・実測の数字**で正面から突き合わせます。どちらがどこまで速いか（そして①がなぜ「速いが読めない」のか）、なぜそうなるか、そして「どっちをいつ選ぶか」。

## 再現手順（手元で動かす）

リポジトリの `parallel/shared-memory/` で：

```sh
make run        # ロック版(P=1..4 一致)/無ロック版(過小集計)/割り込みデモ
```

1. 検証済みの単一ヘッダ Z80 コア（chips `z80.h`、zlib、`NOTICE` 参照）を N 個インスタンス化。
2. 周りに「共有 RAM 窓＋バス調停＋HW セマフォ＋バリア＋IM2 割り込み」のつなぎ層（`mc.c`/`cpu.c`/`machine.h`）を書く。
3. 第1回と同じ `pi_kernel.h` を、共有 RAM の 32bit アキュムレータへ足し込む形に。`USE_LOCK` 有り/無しでビルド。
4. ロック無しで答えが過小に狂い（0 ではない）、ロック有りで第1・2回と `3.14148` で一致するのを確認。割り込みデモは PASS が出る。

「ロックを外すと答えが静かに狂う（しかも 0 ではなく中途半端な値）」を自分の手で、決定論的に再現できます。並行処理の最重要事項を体で確かめる実験として、ちょうどいいかもしれない、と個人的には思っています。
