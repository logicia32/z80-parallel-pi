## はじめに — 1台で出た円周率を、何台かで分けてみる

[第1回](./01.md) で、FPU の無い 8 ビット Z80 に固定小数点で円周率を計算させ、`π ≒ 3.1414` まで出しました。約 131 万クロックかかった、あの計算です。

計算の中身（台形カーネル）は第1回と**同じ 1 ファイル `pi_kernel.h`** を
そのまま使います。今回それを SPMD（全ノード同じ binary・rank で分岐）に
くるみ、ルータ通信を足した別プログラムにします。なので「同じカーネル」
ですが「同じ binary」ではありません——その分、P=1（1 ノード）のクロックは
第1回の素のカーネルより少し増えます（SPMD 分岐とルータ通信ぶん）。

今回は素朴な好奇心の続きです。**あの計算を、Z80 を何台かに分けて分担させたら、速くなるのでしょうか。なるとして、どこまで？** 並列計算（MPI）の教科書でいちばん最初に出てくるのが、まさにこの「円周率を分担して求める」題材なので、ちょうどいい練習だと思って手を動かしてみました。

:::message
**この記事の到達点**：同じ計算を Z80 4 台で分担する「メッセージパッシング型」の並列を組み、台数を 1→2→3→4 と増やしてクロック数を実測。速くなるけれど**頭打ちになる**様子を、数字で見ます。
:::

## 背景 — 「送って持ち寄る」型の並列

並列計算のやり方は、大きく二系統あります。今回はそのうちの一つ、**メッセージパッシング型（分散メモリ型）**。

考え方はシンプルです。各 Z80 は**自分のメモリしか持ちません**。共有メモリはありません。仕事を区間ごとに分け、各台が自分の担当ぶんを計算し、その**部分和を「メッセージで送って」一台に持ち寄り、合算**する。これだけです。

:::message
**メッセージパッシング / 分散メモリ**：計算機どうしがメモリを共有せず、データを「送受信」して協調する方式。スーパーコンピュータの定番 MPI がこの型。対になるのが次回扱う「共有メモリ型」。
:::

:::message
**SPMD（同一プログラム複数データ）**：全ノードに**まったく同じプログラム**を配り、各ノードは自分の番号（rank）で「自分はどこを担当するか」を分岐する書き方。並列の基本形です。
:::

実機の Z80 が 4 台あるわけではないので、Z80 シミュレータ `ucsim_z80` を 4 個起動し、その間を取り持つ小さな仲介プログラム（ホスト側のルータ）を自分で書きました。**このルータが、いわば「ネットワーク」そのもの**になります。遅くて小さい 8 ビットでやる利点はここで、メッセージが 1 通ずつ「見える」。普段は速すぎて見えない通信が、目で追える教材になります。

## やってみる — そして「割り込みは使えない」と分かる

各ノードに rank（0〜3）を配り、担当パネル区間を計算させ、部分和を rank 0 に送り、rank 0 が全部を合算して円周率を出す。素直な構成です。

ここで一つ、最初の発見がありました。受信を**割り込み**で受けたかったのですが、調べていくと、使っていた Z80 シミュレータには「外から割り込みをかける口」がありませんでした。

:::message
**ポーリング / 割り込み**：相手からのデータを受け取る二つの流儀。**ポーリング**は「来てる？ まだ？」と自分から繰り返し確認しに行く。**割り込み**は「来たら向こうが教えてくれる」。一般に割り込みのほうが洗練されていますが、環境次第で使えないこともあります。
:::

割り込みが使えないなら、受信は**ポーリング**でやるしかない。最初は遠回りに感じましたが、これは記事的にはむしろ良くて、**「ポーリングのコスト」がそのまま計測対象**になります（割り込み駆動のほうは、自作する次回でちゃんとやります）。制約を正直に受け入れて前に進む、という普通の判断です。

実装で気をつけたのは、**台形をどう分けるか**。区間の「点」で割って端っこを半分にする素朴なやり方は、境界の重みを数え間違えやすい古典的なバグの巣です。台形（パネル）の単位で分けて足すと、隣り合うパネルが境界点を 1 回ずつ持つので、両端の 1/2 重みが**自動的に正しく**なります。地味ですが、ここを外すと答えが静かにずれます。

## 結果 — 速くなる。が、頭打ちで、しかも「ブレる」

台数 P を 1→4 と変え、ルータ越しに 5 回ずつ実測しました。クリティカル
パス（一番遅い台＝実質の所要時間）の、1 台基準のスピードアップです。

```text
 P=1 : 1,324,206 clks   speedup 1.00x              π=3.14148  （毎回不変）
 P=2 :                  speedup ~1.7x  (1.6〜1.8)   π=3.14148  （走るたび変動）
 P=3 :                  speedup ~2.2x  (1.7〜2.5)   π=3.14148
 P=4 :                  speedup ~2.3x  (1.8〜2.9)   π=3.14148
```

速くはなります。が、結果は**二重に素直でない**。

1. **頭打ち**：理想（2,3,4 倍）に全然届かない。総仕事量（全台合計クロック）
   は台数とともに増えてさえいます。
2. **ブレる**：同じ条件で走らせ直すたびに、P=4 の速度向上が 1.8 倍だったり
   2.9 倍だったり**毎回変わる**（上のカッコ内は観測した散らばりの目安で、
   厳密な上下限ではありません——あなたの環境ではまた違う値が出ます。
   それこそが論点です）。「だいたい 2.3 倍で頭打ち」と一言で言える代物では
   ない。前に書いた版は都合のいい 1 サンプルだけを「結果」として出して
   いました——それは嘘の手前です。実測は散ります。

なぜブレるか。各 rank の**純粋な計算**クロックは決定論的で、走るたびに
1 clk も動きません（例：P=4 の rank1–3 は毎回 321060 / 319924 / 312529）。
動くのは **rank 0 だけ**。rank 0 は自分の計算に加えて「他台の部分和を
受信して合算」する係で、その所要は他台がいつ届くか＝**ホスト側ルータの
スケジューリング**に左右されるからです。

:::message
ここが次回②との対照の芯です。**②（共有メモリ・自作シミュレータ）は
完全に決定論的**で、P=1..4 を何回走らせてもクロックは 1 個も動きません。
①は「ネットワーク役」がホスト上の実プロセスなので本質的に揺らぐ。
「分散は速いが読めない／共有は決定論的だが競合の難しさを抱える」——
この性格差そのものが、最終回の主題になります。
:::

頭打ちの方も、同じ rank 0 が犯人です。各台の計算は台数で N/P に縮むのに、
rank 0 が抱える「受信して合算」は台数を増やしても**縮まない**（むしろ
受け取る相手が増えて重くなる）。**この縮まない直列部分**が全体の足を
引っ張る。これがあの有名な法則の入口です。

:::message
**アムダールの法則**：処理の中に「並列にできない直列部分」があると、いくらコアを増やしても速度には上限がある、という法則。今回の rank 0 の合算がまさにその直列部分です。次回・次々回で、この法則を数字でもっとはっきり見ます。
:::

もう一つ、学生さん向けに面白いと私が感じた点を。**π はどの台数でも 3.14148 で、1 ビットも変わりませんでした**。固定小数点（整数）の足し算は順序を変えても結果が同じ＝**決定論的**だからです。浮動小数点の総和だと、足す順番で最後の桁がふらつくことがある。「整数で持つと再現性が高い」という、地味だけど実務で効く性質が、目の前の数字で確認できます。精度が台数で変わらないのも、第1回で見た「精度の壁は語長」の裏返しです。

## 詰まった所

- **割り込みが使えない**のは環境の制約でした。隠さず受け止めて「ポーリングのコストを測る」に切り替えたほうが、結果として記事の芯ができました。制約は、扱い方しだいで素材になります。
- **ルータ（ネットワーク役）が地味に主役**でした。並列の難しさの多くは計算ではなく「つなぎ」にある、というのを小さく追体験できます。
- 台形の**パネル単位分割**。点で割って半分、をやると静かに間違える。並列化したコードの検算は、まず逐次版と**ビット一致**するかを見る——これが効きました。

## まとめと、次回

- 同じカーネルを 4 台に分けると速くはなる。が **理想に遠く届かず、しかも
  走るたびに大きく振れる**（P=4 で実測 1.8〜2.9 倍、毎回別の値。これは
  目安で厳密境界ではない）。rank 0 の合算という縮まない直列部分
  （アムダール）＋ ホスト依存の揺らぎ。
- 分けても答えは変わらない（固定小数点の決定論。π=3.14148 で 1 ビットも
  動かない）。精度も台数で変わらない（語長の壁）。
- 「制約を正直に受け入れる」も、「都合のいい 1 サンプルを結果と言わない」
  のも、立派な設計判断であり、書き手の誠実さです。

次回（第3回）は、まったく別のやり方——**皆で 1 つのメモリを共有する**型——を、シミュレータごと自分で作ってみます。共有 RAM、バス調停、そして今回できなかった**本物の割り込み（IM2・手書き ISR）**。「ロックを忘れると答えが静かに狂う（ロストアップデート）」という、並行処理でいちばん大事な事故も、実測の数字で見せます。

## 再現手順（手元で動かす）

コードと手順はリポジトリにあり、README が「再現マニュアル」です
（`pi_kernel.h` は第1回と同じ 1 ファイル）。流れだけ：

1. 第1回と同じ `pi_kernel.h` を SPMD でくるむ（rank で担当区間を分岐）。
2. `ucsim_z80` を台数ぶん起動し、間をつなぐ小さなルータ（数十行）を用意。
3. 部分和を rank 0 に集めて合算。台数 P を変えて実行。
4. 各台の停止後、シミュレータの `state` でクロック数を読む。

P を 1→4 と変えると、スピードアップが理想線から離れていくのが自分の目で見えます。「アムダールの法則を体感する」小さな実験として、ちょうどいいかもしれない、と個人的には思っています。
